# Integration Surfaces — Procurement / Adapter Readiness Map

Last checked: 2026-04-25

Scope: a procurement-style readiness sheet for the seller and developer. For each surface where a "patient/family voice" slot does or might exist, what is the realistic adapter path *this weekend* and what is the realistic adapter path *this year*. Where a fact cannot be confirmed in public sources it is marked as such.

## Q1 — Nedap Caren / Ons Dossier *Mijn verhaal*

### Readiness card

- **Surface.** Nedap *Ons Dossier* (professional ECD) + *Caren* (family/client portal). Slot: *Persoonlijk cliëntverhaal* (visible in Caren as *Mijn verhaal*). Free-text + photo, surfaced under the client's name in Ons Dossier and in Caren ([Nedap support — Persoonlijk cliëntverhaal](https://support.nedap-ons.nl/support/solutions/articles/103000265791-persoonlijk-cli%C3%ABntverhaal-in-ons-dossier-en-caren), [Caren support — Mijn Verhaal](https://support.carenzorgt.nl/support/solutions/articles/75000013508-mijn-verhaal)).
- **Adapter path.** **No public API for *Mijn verhaal* is documented.** The Nedap Ons API does exist as a partner-mediated programme — external apps can "retrieve, place and update specific data" against opened endpoints, gated on a healthcare-organisation commitment ([Ons-API.nl — what is Ons API](https://ons-api.nl/), [Ons-API.nl — integration partners](https://ons-api.nl/Integratiepartners.html)). The published partner list is small (Creq, Quattri, Infozorg) and the partner programme appears curated rather than an open marketplace. Whether the *Persoonlijk cliëntverhaal* field is exposed in any opened endpoint is **not confirmed in public sources** — it is not mentioned in the API koppelingen support article ([Nedap support — API-koppelingen](https://support.nedap-ons.nl/support/solutions/articles/103000273959-api-koppelingen)) nor on ons-api.nl. Activation of *Mijn verhaal* itself is gated by a Caren-support ticket; the customer org cannot self-enable it.
- **Data shape.** Free text + photo + structured sub-fields for *aanspreekvorm* (how I want to be addressed) and a list of *belangrijke voorkeuren* (key preferences). Not a published schema; not FHIR-typed.
- **Gating.** Two role-permission tasks: *Persoonlijk cliëntverhaal inzien* and *Persoonlijk verhaal van een cliënt wijzigen*. The client (or representative) owns the content; the provider organisation activates the slot.
- **Editing-disabled state.** Still current as of the support article: *"Het wijzigen van het cliëntverhaal staat op dit moment uit omdat deze functie niet naar behoren werkt."* Sharing works, editing is off pending a fix ([Nedap support — Persoonlijk cliëntverhaal](https://support.nedap-ons.nl/support/solutions/articles/103000265791-persoonlijk-cli%C3%ABntverhaal-in-ons-dossier-en-caren)). This is an opportunity, not a blocker — the slot is in market but not delighting users.
- **Realistic v1 move.** Render a structured PDF/markdown export shaped like the *Mijn verhaal* fields (aanspreekvorm + voorkeuren + free-text narrative + photo placeholder), and have the family or casemanager paste it into Caren. No programmatic write in v1. Approach a Nedap integration partner (Creq is already a "trots integratiepartner" of Nedap) to scope a partner-mediated write later.
- **Sources.** [Nedap support — Persoonlijk cliëntverhaal](https://support.nedap-ons.nl/support/solutions/articles/103000265791-persoonlijk-cli%C3%ABntverhaal-in-ons-dossier-en-caren), [Caren support — Mijn Verhaal](https://support.carenzorgt.nl/support/solutions/articles/75000013508-mijn-verhaal), [Sensire handleiding](https://www.sensire.nl/getmedia/0ef718c9-7603-4640-b3c5-9da2e1e093a7/Handleiding-Mijn-Verhaal-bij-CarenZorgt.pdf), [Ons-API.nl](https://ons-api.nl/), [Ons-API integration partners](https://ons-api.nl/Integratiepartners.html), [Nedap support — API koppelingen](https://support.nedap-ons.nl/support/solutions/articles/103000273959-api-koppelingen), [Nedap implementatiepartners](https://nedap-ons.nl/starten-met-nedap-ons/implementatiepartners/), [Creq Nedap-partner](https://creq.nl/nedap-partner/).

## Q2 — PinkRoccade *mijnCaress* / Ecare / Pluriform: family portal and life-story slot

### Readiness card

- **Surface.** PinkRoccade *mijnCaress* (ECD for VVT and gehandicaptenzorg) + *mijnCaress Cliëntportaal* (the family/client-facing portal). Branded under "mijnCaress Cliëntportaal" — there is no separate *Familienet* / *MedZorg* product in the PinkRoccade lineup that could be confirmed. Cases such as BrabantZorg and Liberein deploy the Cliëntportaal directly to mantelzorgers ([BrabantZorg case](https://pinkroccade-healthcare.nl/case/brabantzorg-mijncaress-clientportaal-schromen-gewoon/), [Liberein case](https://pinkroccade-healthcare.nl/case/liberein-mijncaress-clientportaal/)).
- **Life-story slot.** mijnCaress includes *zorgleefplan*, *persoonsbeeld* and intake reports; the M&I/Partners 2025 ECD inventarisatie confirms intake-to-persoonsbeeld-to-zorgplan as a core flow ([M&I Partners — ECD-inventarisatie 2025](https://mxi.nl/kennis/730/ecd-inventarisatie-2025-gehandicaptenzorg-en-ouderenzorg)). A dedicated *levensverhaal* module under that exact name is **not confirmed in public materials**; the persoonsbeeld field is the closest surface and is the de-facto life-story container in PinkRoccade ECDs.
- **Adapter path.** Better than Nedap. mijnCaress publicly markets an **API-first / REST architecture on Spring Boot**, with the Cliëntportaal and Cliëntplanning built on those APIs, and explicit positioning around MedMij/PGO interoperability ([mijnCaress — wordt steeds meer Open](https://www.mijncaress.nl/mijncaress-wordt-steeds-meer-open/), [mijnCaress 5.4 — ZIBs en iStandaarden](https://www.mijncaress.nl/mijncaress-5-4-zibs-clientplanning-en-istandaarden), [mijnCaress — streven naar interoperabiliteit](https://www.mijncaress.nl/streven-naar-interoperabiliteit)). Integration partners named publicly: Boomerweb, Medimo, NCare, Ysis, ZorgDomein, Point. Whether persoonsbeeld is writeable via that API today is **not confirmed in public sources** — likely a partner conversation rather than a published endpoint.
- **Data shape.** Moving toward zibs / iStandaarden alignment from v5.4 onward. Not yet FHIR-on-the-wire for persoonsbeeld; structured but proprietary.
- **Gating.** Provider-org account for write; client/mantelzorger account for the Cliëntportaal read. mijnCaress 4.3+ exposes care-plan content into the portal so the data round-trip exists.
- **Realistic v1 move.** Same as Nedap — structured export PDF/markdown that maps to *persoonsbeeld* fields, pasted by family or casemanager. Strategically, mijnCaress is the more API-friendly second target: their explicit MedMij/PGO/ZIB framing means a FHIR-wrapped output is more likely to land cleanly.
- **Sources.** [mijnCaress — wordt steeds meer Open](https://www.mijncaress.nl/mijncaress-wordt-steeds-meer-open/), [mijnCaress 5.4](https://www.mijncaress.nl/mijncaress-5-4-zibs-clientplanning-en-istandaarden), [mijnCaress Platform](https://mijncaress.nl/oplossingen/mijncaress-platform/), [mijnCaress 4.6 highlights](https://issuu.com/pinkroccade/docs/highlights_mijncaress_4.6), [BrabantZorg case](https://pinkroccade-healthcare.nl/case/brabantzorg-mijncaress-clientportaal-schromen-gewoon/), [M&I/Partners ECD-inventarisatie 2025](https://mxi.nl/kennis/730/ecd-inventarisatie-2025-gehandicaptenzorg-en-ouderenzorg).

## Q3 — MedMij PGOs: national schema for life-story / care preferences / ACP

### Readiness card

- **Surface.** MedMij is the Dutch national framework for PGOs. Catalogus is published by Stichting MedMij ([MedMij Catalogus — actueel](https://catalogus.medmij.nl/overzicht/actueel/)) with a standard XML release (most recent reviewed: Release 4 / sequence 50 / April 2026).
- **Life-story / preferences gegevensdienst.** **Does not exist in the current catalogue.** A direct review of *actuele gegevensdiensten* (medication, vaccinations, lab, BgZ, GP data, GGZ, vitals, allergies, documents, appointments, questionnaires, long-term care, pregnancy) and *gegevensdiensten in ontwikkeling* (~21 alpha/beta services, mostly clinical: vitals, dental care, encounters, demographics, woonsituatie) returned **no levensverhaal, persoonsbeeld, zorgvoorkeuren, ACP or proactieve-zorgplanning service** ([Actuele gegevensdiensten](https://catalogus.medmij.nl/overzicht/actueel/actuele-gegevensdiensten), [Gegevensdiensten in ontwikkeling](https://catalogus.medmij.nl/overzicht/actueel/gegevensdiensten-in-ontwikkeling)).
- **PZP information standard (separate from MedMij catalogue).** Nictiz publishes the **Informatiestandaard Proactieve Zorgplanning (PZP)** — explicitly the Dutch ACP standard, IKNL-developed (2023), built on **zibs2017 / zibs2020** with **FHIR R4 and STU3 implementation guides**. Status April 2026: ReleaseCandidate.2, valid from 2026-03-03; not production-stable, not yet a MedMij gegevensdienst. Conceptual scope matches our six wedges almost exactly: *doelen, voorkeuren, wensen, grenzen, waarden, behoeften* of the patient ([Nictiz — Informatiestandaard PZP](https://nationalebibliotheek.nictiz.nl/bibliotheek/proactieve-zorgplanning-pzp/), [Nictiz — consultatie PZP](https://nictiz.nl/nieuws/consultatie-proactieve-zorgplanning/), [PZP v0.1.3 Beta3](https://nationalebibliotheek.nictiz.nl/releases/informatiestandaard-proactieve-zorgplanning-pzp-v0-1-3-beta3/)).
- **PGO support today.** Ivido, Drimpy, Quli are MedMij-labelled ([ICT&health — fifth PGO](https://icthealth.nl/nieuws/vijfde-pgo-aanbieder-krijgt-medmij-label/), [ICT&health — first PGOs](https://www.icthealth.nl/nieuws/eerste-pgos-krijgen-medmij-label-uitgereikt)). **No MedMij-label PGO publicly advertises a structured life-story or preference-capture surface** that we could confirm in public materials. PGO.nl lists certified vendors but does not enumerate per-vendor feature support for biographical data.
- **Vendor-neutral distribution.** Not yet. Without a MedMij gegevensdienst for life story / care preferences, MedMij-conformant output does not get us automatic distribution to PGOs. The *strategic* path is: structure our output today on the **PZP zibs / FHIR profiles** so we are pre-aligned with the standard most likely to become the MedMij gegevensdienst for ACP-shaped data, and lobby to be the reference implementation when PZP graduates from RC to production and gets a MedMij wrapper.
- **Realistic v1 move.** Generate a FHIR R4 PZP-shaped resource bundle as a parallel export format alongside the human-readable PDF/markdown. We won't ship to a PGO this weekend but the *shape* is the procurement story.
- **Sources.** [MedMij Catalogus](https://catalogus.medmij.nl/overzicht/actueel/), [MedMij — informatiestandaarden overview](https://medmij.nl/informatiestandaarden/overzicht-van-medmij-gegevensdiensten/), [Nictiz PZP standard page](https://nationalebibliotheek.nictiz.nl/bibliotheek/proactieve-zorgplanning-pzp/), [Nictiz — consultatie PZP](https://nictiz.nl/nieuws/consultatie-proactieve-zorgplanning/), [Pgo.nl](https://www.pgo.nl/), [Ivido](https://ivido.nl/), [Drimpy — PGO uitleg](https://www.drimpy.com/persoonlijke-gezondheidsomgeving-pgo-wat-is-dat/).

## Q4 — Hospital ECDs (HiX, Epic NL, Nexus): is there a patient-voice slot?

### Readiness card

- **Surface.** ChipSoft *HiX* (dominant NL hospital EPD), Epic NL, Nexus. HiX has a *HiX Patient* portal app and a geriatrics module. The geriatrics module supports VMS/GRP screening of patients ≥70 and predefined anamnesis forms (cognitive, falls/mobility, medication review) ([ChipSoft — Geriatrie](https://www.chipsoft.com/nl-NL/hix-abc/artikel/30), [HiX Patient — App Store](https://apps.apple.com/nl/app/hix-pati%C3%ABnt/id1605109373)).
- **Patient-voice / personhood / "this is me" slot.** **Not confirmed in public sources for any of HiX, Epic NL, or Nexus.** No public ChipSoft documentation describes a dedicated personhood/preferences/levensverhaal field for dementia or frail-elderly patients. Hospital admission is therefore **a standalone-UI moment for our product, not an adapter moment**, until or unless we partner with a hospital that has built a custom HiX form.
- **Cross-vendor exchange.** HiX↔Epic↔Nexus exchange runs on FHIR via the national *(Zorg)platform*; the exchange is BgZ-shaped (clinical summary), not personhood-shaped ([ICT&health — HiX/Epic exchange](https://www.icthealth.nl/nieuws/ziekenhuizen-wisselen-patientdata-uit-tussen-epd-hix-en-epic), [ChipSoft — landelijk Zorgplatform](https://www.chipsoft.com/nl-NL/nieuws/835)).
- **Dementia-friendly hospital initiatives.** A 2025 NGN working-group programme on *Dementievriendelijke Ziekenhuiszorg* is active; the *Kennissynthese ziekenhuiszorg dementie* maps gaps and explicitly calls for person-centered approaches and informal-caregiver support, but does not (yet) prescribe a data field in the EPD ([Geheugenpoliklinieken — kennissynthese](https://www.geheugenpoliklinieken.nl/actueel/nieuws/kennissynthese-ziekenhuiszorg-dementie), [Zorgvisie — elf handvatten](https://www.zorgvisie.nl/elf-handvatten-voor-dementievriendelijke-ziekenhuiszorg/)). This is the policy thread to ride for hospital adoption.
- **Adapter path.** None today. eOverdracht (the nursing handover standard from VVT into hospitals) does carry social/living-condition zibs and *advance directives* alongside clinical content ([Nictiz eOverdracht v4.0](https://informatiestandaarden.nictiz.nl/wiki/vpk:V4.0_Ontwerp_eOverdracht), [V&VN — eOverdracht factsheet](https://www.venvn.nl/media/megjb1mo/20200824-factsheet-eoverdracht.pdf)) — that gives us a route *into* the hospital via the VVT side rather than a direct HiX integration. By end-2026, every healthcare professional must be able to work with eOverdracht ([V&VN — verpleegkundige overdracht](https://www.venvn.nl/thema-s/vastleggen-en-overdragen/het-overdragen-van-patientgegevens/verpleegkundige-overdracht)).
- **Data shape.** eOverdracht is FHIR + zibs. Our PZP-aligned output sits naturally inside an eOverdracht packet.
- **Realistic v1 move.** Standalone UI ("ward-handover view") accessible via a QR code on the patient's wristband or family-facing card. No HiX integration in scope.
- **Sources.** [ChipSoft — Geriatrie](https://www.chipsoft.com/nl-NL/hix-abc/artikel/30), [HiX Patient](https://www.chipsoft.com/nl-be/hix-abc/hix-abc-articles/hix-patient/), [ICT&health — HiX/Epic exchange](https://www.icthealth.nl/nieuws/ziekenhuizen-wisselen-patientdata-uit-tussen-epd-hix-en-epic), [Geheugenpoliklinieken — kennissynthese](https://www.geheugenpoliklinieken.nl/actueel/nieuws/kennissynthese-ziekenhuiszorg-dementie), [Zorgvisie — elf handvatten](https://www.zorgvisie.nl/elf-handvatten-voor-dementievriendelijke-ziekenhuiszorg/), [Nictiz eOverdracht v4.0](https://informatiestandaarden.nictiz.nl/wiki/vpk:V4.0_Ontwerp_eOverdracht), [V&VN factsheet eOverdracht](https://www.venvn.nl/media/megjb1mo/20200824-factsheet-eoverdracht.pdf).

## Q5 — VVT-side recommended schema

### Readiness card

- **Vilans — Persoonsgerichte Zorg.** Vilans publishes the *thema* and methodology pages on persoonsgerichte zorg and an article *"Persoonsgerichte zorg op basis van data in ouderenzorg"* — but **no normative schema**, no recommended field set, no zib mapping that an ECD-neutral product is told to align with ([Vilans — persoonsgerichte zorg](https://www.vilans.nl/thema-s/persoonsgerichte-zorg), [Vilans — ECD](https://www.vilans.nl/kennis/ecd-elektronisch-clientendossier)). Confirms persoonsbeeld and levensverhaal as concepts to capture, does not specify how.
- **Waardigheid en Trots.** Publishes *Methodiek Persoonsgerichte Zorg* and the *Verhaalcirkel*; the WenT article on ECDs and quality of life uses NANDA / NOC / NIC for problem-and-outcome structuring rather than a personhood schema ([WenT — methodiek persoonsgerichte zorg](https://www.waardigheidentrots.nl/tools/methodiek-persoonsgerichte-zorg/), [WenT — ECD kwaliteit van leven](https://www.waardigheidentrots.nl/praktijk/ecd-kwaliteit-van-leven-en-efficient-werken/)). Not a data schema.
- **Zorginstituut — Kwaliteitsstandaard Levensvragen.** Exists, addresses spiritual/meaning questions in care; conceptual rather than data-modelled ([Zorginzicht — Kwaliteitsstandaard Levensvragen PDF](https://www.zorginzicht.nl/binaries/content/assets/zorginzicht/kwaliteitsinstrumenten/Kwaliteitsstandaard+Levensvragen.pdf)).
- **V&VN — eOverdracht.** The closest thing to a national VVT data schema for non-clinical patient context. v4.0 is the *minimale eOverdracht*; sections include *algemene leefomstandigheden, sociale omstandigheden, wilsverklaringen* — but no levensverhaal zib and no preference-profile zib ([Nictiz eOverdracht v4.0](https://informatiestandaarden.nictiz.nl/wiki/vpk:V4.0_Ontwerp_eOverdracht), [V&VN secties eOverdracht](https://www.venvn.nl/media/an4iaehg/secties-eoverdracht.pdf)).
- **Nictiz — PZP information standard** (covered in Q3) is the **best candidate for the schema we should align with** today: explicitly captures preferences/wishes/values/boundaries, zib-based, FHIR-typed, RC-stage, owned by Nictiz/IKNL, aligned with the Leidraad Proactieve Zorgplanning that NHG/Verenso/V&VN/NVAVG/Patiëntenfederatie all signed.
- **InZicht / VIPP.** *VIPP InZicht* is the VWS subsidy that has been pushing VVT/GHZ providers to implement eOverdracht and to disclose data to PGOs by end-2026 ([Beter Healthcare — VIPP InZicht](https://beterhealthcare.nl/vipp-inzicht/), [Drimpy — VIPP InZicht](https://www.drimpy.com/projecten/vipp-inzicht-wat-is-dat-en-waarom/)). It funds plumbing, not new schemas — but the political momentum is the lever to argue "the patient-voice gegevensdienst is the missing brick."
- **Bottom line.** **There is no published national schema for *persoonsbeeld* / *levensverhaal* / *zorgvoorkeuren*.** The vacuum is real. The closest standards-grade adjacent thing is **Nictiz PZP**, which we should adopt as our internal data shape and pitch externally as "we ship PZP-conformant output today, before any ECD does."
- **Sources.** [Vilans — persoonsgerichte zorg](https://www.vilans.nl/thema-s/persoonsgerichte-zorg), [Vilans — ECD](https://www.vilans.nl/kennis/ecd-elektronisch-clientendossier), [WenT — methodiek persoonsgerichte zorg](https://www.waardigheidentrots.nl/tools/methodiek-persoonsgerichte-zorg/), [WenT — ECD kwaliteit van leven](https://www.waardigheidentrots.nl/praktijk/ecd-kwaliteit-van-leven-en-efficient-werken/), [Kwaliteitsstandaard Levensvragen](https://www.zorginzicht.nl/binaries/content/assets/zorginzicht/kwaliteitsinstrumenten/Kwaliteitsstandaard+Levensvragen.pdf), [Nictiz PZP](https://nationalebibliotheek.nictiz.nl/bibliotheek/proactieve-zorgplanning-pzp/), [VIPP InZicht — Beter Healthcare](https://beterhealthcare.nl/vipp-inzicht/), [VIPP InZicht — Drimpy](https://www.drimpy.com/projecten/vipp-inzicht-wat-is-dat-en-waarom/).

## Prioritised integration roadmap — next 6 months

1. **Nedap *Mijn verhaal* (paste-in v1, partner-mediated write v2).** Highest distribution leverage in NL VVT and home-care; the slot is in market and currently underused (editing disabled). Move first because procurement officers already know the field name. Path: ship the structured-export today, open conversations with Creq / Quattri / Infozorg, target a partner-mediated write within the InZicht funding window.
2. **PinkRoccade mijnCaress *persoonsbeeld* (FHIR-friendly v1).** Second VVT vendor, more API-friendly. Use the explicit ZIB/iStandaarden v5.4 framing to push a FHIR-PZP-shaped write into persoonsbeeld. Likely faster to a real API write than Nedap.
3. **Nictiz PZP reference implementation.** Adopt PZP zibs/FHIR as our internal data shape. As PZP graduates from RC to production, position to be the reference implementation and the canonical authoring tool. This is the lever that turns a "yet another startup" pitch into "we are shipping the standard."
4. **MedMij PGO export (Ivido / Drimpy / Quli).** Ship a FHIR-PZP bundle as a MedMij-style download today; lobby for a *Levensverhaal en Zorgvoorkeuren* gegevensdienst. Not until 2027 realistically.
5. **eOverdracht handoff into hospitals.** Add the patient-voice content to outbound eOverdracht packets so hospitals receive it whether or not HiX has a slot. Quietly the most scalable way into hospitals.
6. **Hospital-direct (HiX, Epic, Nexus).** Standalone UI / QR-on-wristband mode only. Revisit when a dementia-vriendelijke ziekenhuiszorg pilot site asks.

## Three realistic adapter paths for this weekend's demo

1. **Structured PDF + markdown export shaped to *Mijn verhaal*** (aanspreekvorm, voorkeuren, narrative, photo placeholder), pasted by family or casemanager into Caren. Reads as "fills the field Nedap already ships."
2. **FHIR R4 PZP-shaped resource bundle**, downloadable from the standalone UI. Reads as "MedMij-grade output today, pre-aligned with the Nictiz Proactieve Zorgplanning standard."
3. **Standalone web view with QR-code share link**, gated by patient/representative consent — covers the hospital admission, locum nurse, and family-without-portal cases. Reads as "works on day one, in any setting, no integration required."

## Policy / standards thread to ride

**Nictiz Informatiestandaard Proactieve Zorgplanning (PZP)** is the single most leveraged thread. It is FHIR-R4, zib-based, owned by Nictiz/IKNL, signed by the major professional bodies, currently RC-stage, and conceptually one-to-one with our six wedges. If we ship PZP-conformant output now and become the de-facto authoring tool while the standard is still hardening, we get vendor-neutral distribution faster than chasing each ECD. The companion lever is **VIPP InZicht / end-2026 eOverdracht mandate** — we ride the funding that is already pushing VVT providers to expose structured data, and we argue that the patient-voice gegevensdienst is the obvious next brick MedMij is missing.
